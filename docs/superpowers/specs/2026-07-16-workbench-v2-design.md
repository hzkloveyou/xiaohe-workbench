# 小贺的工作台 V2 多页面升级设计

## 1. 背景与目标

本次工作是在现有“小贺的工作台”上增量升级，不重新创建项目，不替换 React、TypeScript、Vite、Dexie、Cloudflare Worker/D1、GitHub Pages 等既有技术栈，也不删除现有搜索、书签、今日专注、主题、备份、登录、恢复码、同步和 PWA 能力。

升级采用“方案 B：多页面工作台＋统一指令中心”，一次发布完成四个相互衔接的阶段：

1. 全局指令中心。
2. 万能收集箱与书签增强。
3. 今日计划 2.0 与专注统计。
4. GitHub 面板、状态中心和外部集成。

核心目标是把现有启动台升级为高频、可靠、适合桌面和手机使用的个人控制中心，同时保持清新、克制的“晨雾玻璃”视觉风格和本地优先的数据模型。

## 2. 现状与约束

### 2.1 现有能力

- React 19、TypeScript、Vite 静态前端。
- Dexie/IndexedDB 本地实体和待同步队列。
- Cloudflare Worker、D1、Cookie 会话和增量同步。
- Motion 动效与 `prefers-reduced-motion` 支持。
- dnd-kit 鼠标、触摸和键盘拖拽。
- GitHub Pages 构建发布，Cloudflare Worker Route 提供自定义域名。
- 游客本地使用、账户同步、备份导入导出和离线应用壳。

### 2.2 必须修复的既有边界

- 同步引擎在本机没有待上传变更时不会主动拉取远端数据，导致新设备登录后可能拿不到已有工作台内容。
- 同步请求没有完整的批量、游标拉取和受控重试策略。
- `App.tsx` 已承担过多领域编排，无法安全容纳本次四个阶段。
- 搜索引擎和部分界面偏好没有统一进入可同步数据。
- 任务和计时器模型不足以支持计划、重复任务和历史统计。

### 2.3 不可违反的约束

- 浏览器继续承担 310,000 次 PBKDF2-SHA-256 密码证明派生；Worker 不执行昂贵密码派生。
- 不新增 `public/CNAME`；根域名继续由 Cloudflare Worker Route 代理 GitHub Pages 项目路径。
- 不将 GitHub、Cloudflare 或第三方私人令牌写入前端、仓库、备份、日志或 D1 工作台实体。
- 数据写入仍然先落本地；网络或账户故障不得阻止游客使用。
- 旧书签、任务、便笺、主题和备份必须向后兼容。

## 3. 产品信息架构

应用使用共享外壳和四个主要页面。桌面端使用顶部页面导航；手机端使用固定底部导航。页面切换保留清晰 URL，并支持浏览器前进、后退、直接刷新和离线访问。

### 3.1 概览页 `/`

概览页保留原有品牌、日期、问候和主要搜索体验，并改为信息摘要：

- 全局指令中心入口。
- 今日三件事与最近一项专注进度。
- 收藏及最近使用书签。
- 万能收集快捷入口和待整理数量。
- 今日专注时长及完成任务数。
- 网站/API 状态摘要。
- GitHub 最近仓库或部署摘要。

概览页不承载完整管理表单，避免恢复为无限向下延伸的单页仪表盘。

### 3.2 收集页 `/collect`

收集页包含万能收集箱与完整书签管理。

万能输入根据显式类型、指令前缀和内容形态判断为链接、记录或任务：

- URL 或裸域名：链接。
- `任务`、`todo` 等前缀：任务。
- 其他文本：记录。

用户始终可以在保存前手动切换类型，自动判断不能剥夺控制权。任务只识别“今天、明天、后天”和 `YYYY-MM-DD` 四类确定性日期表达，无法识别的文字保留在标题中。收集项支持待整理、归档、转换和删除。链接元数据读取由受限 Worker 预览接口提供；获取失败时使用域名和文字首字，不阻止保存。

书签管理支持：

- 分组、多个标签、收藏。
- 标题、网址、标签和域名搜索。
- 收藏、最近使用、访问次数和分组筛选。
- 编辑、删除、复制链接、拖动排序。
- 重复链接提示。
- 待整理链接转为正式书签。

### 3.3 今日页 `/today`

今日页把现有“三件事＋计时器”扩展为轻量计划系统：

- 今天、明天、以后和已完成视图。
- 标题、日期、优先级、排序和可选备注。
- 新增、编辑、删除、完成和恢复。
- 每日、每周、每月重复；完成当前实例时生成下一实例。月度重复遇到不存在的日期时使用该月最后一天。
- 今日最重要的三件事继续作为概览摘要，而不是任务总量上限。
- 25、45、60 分钟专注时长。
- 可把专注会话关联到一个任务。
- 完成会话写入独立历史实体。
- 最近七天专注时长、会话数、任务完成数和连续专注天数。

统计图使用轻量 SVG/CSS，不引入完整图表库。

### 3.4 连接页 `/connect`

连接页包含 GitHub、本站运行状态和外部快捷入口。

GitHub 面板首版默认读取公开用户 `hzkloveyou` 的公开数据：

- 最近更新仓库。
- 仓库 Stars、主要语言和更新时间。
- `xiaohe-workbench` 最近提交和 Actions 运行摘要。
- 手动刷新、十五分钟缓存和上次更新时间。
- 请求失败、限流或离线时优先显示未过期或最近缓存。

不在浏览器保存 GitHub 私人令牌，因此首版不展示私有仓库、私人通知或需要授权的个人数据。

状态中心检查固定的项目服务：

- `https://080492.xyz`。
- `https://api.080492.xyz/health`。
- GitHub 仓库或发布状态。

检查在进入页面或手动刷新时执行，不做高频后台轮询。外部快捷入口复用书签能力，不另建一套重复链接模型。

## 4. 全局指令中心

在任意页面按 `Ctrl/Command + K` 或点击导航入口打开指令中心。指令中心提供统一检索和动作执行：

- 搜索书签、标签和页面。
- 打开工作台页面。
- 执行网页搜索。
- 添加链接、记录或任务。
- 启动指定时长的专注会话。
- 切换主题。
- 打开 GitHub 或状态中心。

结果按“建议、页面、书签、动作、网页搜索”分组。键盘支持上下方向键、Enter 和 Escape；弹层打开时焦点被约束，关闭后返回触发元素。移动端使用全屏面板，桌面端使用居中命令面板。

指令匹配采用确定性字符串和关键词规则，不引入在线 AI。没有匹配时提供网页搜索和快速收集两个明确的兜底动作。

## 5. 应用架构

### 5.1 应用外壳与路由

新增共享应用外壳，负责品牌、桌面导航、手机底部导航、账户入口、指令中心和全局 Toast。页面组件只负责本领域展示与交互。

使用成熟客户端路由库和 Browser History 路由。Cloudflare 站点代理对 HTML 导航请求提供 SPA 回退；静态资源、API 和真实 404 不被回退吞掉。页面级代码采用懒加载，避免四个页面全部进入初始包。

### 5.2 工作台状态边界

从 `App.tsx` 抽出工作台 Provider/控制器，集中负责：

- 初始化 Dexie。
- 读取和刷新快照。
- 应用本地实体变更。
- 当前账户和同步状态。
- 主题与偏好。
- 全局错误与非阻塞通知。

书签、收集、计划、专注、GitHub 和状态模块通过明确的领域函数使用工作台，而不是直接操作 Dexie 或互相导入组件内部状态。

### 5.3 数据模型

在保持旧实体可解析的前提下扩展实体类型：

- `bookmark`：增加可选 `tags`、`favorite`、`visitCount`、`lastVisitedAt`。
- `task`：增加可选 `scheduledFor`、`dueAt`、`priority`、`note`、`recurrence`、`seriesId`、`completedAt`。
- `inboxItem`：保存待整理的链接、记录或任务原始输入及解析结果。
- `focusSession`：保存开始、结束、计划时长、实际时长、关联任务和完成状态。
- `preference`：保存搜索引擎、页面偏好、GitHub 用户名等非敏感设置。

不需要新增 D1 业务表；现有 `workspace_entities` 可以保存新的类型和 JSON 数据。共享 Zod 契约必须在 Worker 发布前后兼容新旧字段。

备份格式继续接受现有版本，并把新增实体包含在导出中。导入前完整校验，失败时不覆盖当前数据。

### 5.4 同步修复

同步周期统一为：

1. 在线且已登录时先推送待处理变更，按不超过 API 上限的批次发送。
2. 无论本地是否有待上传数据，都从已保存游标拉取远端变更。
3. 合并远端实体并保存新游标。
4. 页面可见、重新联网、登录成功和受控定时器触发同步。
5. 网络失败采用有上限的指数退避；认证失败停止重试并提示重新登录。

新设备首次登录从游标零开始拉取。删除标记继续参与时钟比较，避免已删除实体复活。

## 6. 外部数据与安全

### 6.1 GitHub

优先使用 GitHub 公开 API，不使用私人 Token。客户端限制刷新频率并缓存非敏感公开响应。展示数据时进行运行时校验；字段缺失不能导致整个页面崩溃。

### 6.2 链接元数据

链接保存不依赖元数据接口。服务端提供受限预览接口：只允许 HTTP/HTTPS，拒绝本机名、私网和 IP 字面量，限制重定向、响应大小、内容类型和超时，并只返回标题、描述、站点名等必要字段。任何预览错误均降级为手动标题和本地图标。

### 6.3 状态检查

只检查项目内固定服务，不提供任意 URL 的服务端探测器，避免 SSRF。跨域健康接口只开放只读响应所需的最小 CORS。

## 7. 视觉与响应式

继续使用现有天空蓝、薄荷绿、奶油白、柔和玻璃卡片和低对比边框。升级重点是信息层级，不重做品牌。

- 桌面：顶部导航，内容最大宽度保持一致，页面使用 12 列或领域内双栏布局。
- 平板：导航保留，卡片降为 4–8 列。
- 手机：底部导航；页面单列；指令中心全屏；主要操作按钮至少 44px。
- 所有页面都有加载、空、错误、离线和数据过期状态。
- 动画只用于页面切换、弹层和布局变化；减少动态效果时关闭非必要动画。
- 不出现横向滚动；底部导航为安全区预留空间。

## 9. 测试策略

所有功能以失败测试开始，并覆盖以下层次：

- 纯函数：指令匹配、收集分类、书签筛选、任务重复、专注统计、GitHub 响应解析。
- 仓储与同步：旧数据兼容、首次拉取、批量推送、游标、离线队列、认证失效。
- 组件：键盘指令中心、收集表单、书签筛选、任务编辑、统计空状态、GitHub 缓存与错误状态。
- Worker：新实体契约、健康 CORS、站点子路由 SPA 回退与静态 404。
- 可访问性：页面导航、弹层焦点、表单名称、状态公告和减少动态效果。
- E2E：四页切换、深链接刷新、移动端无横向滚动、离线壳、数据刷新持久化、原有书签与主题流程。
- 生产验收：根页面、子页面、静态资源、API、登录、跨设备同步、恢复、PWA、GitHub 降级和状态中心。

## 10. 发布顺序与回滚

1. 从当前通过基线检查的代码开始，按领域测试驱动实现。
2. 完成前端、共享契约和 Worker 测试。
3. 先部署兼容新实体的 API Worker；本次预期不需要 D1 迁移，但仍检查远端迁移状态。
4. 部署支持 SPA 回退的站点 Worker。
5. 推送 `main`，由 GitHub Pages 构建并发布前端。
6. 验证线上准确提交、四个页面、静态资源和真实用户流程。
7. 生产验证完成后更新 `C:\Users\Tom\Desktop\project` 备份和 AI 交接材料。

回滚时前端回到上一已验证提交；Worker 独立回滚到上一版本。因为新增字段和实体对旧前端可忽略，且不做破坏性 D1 迁移，回滚不删除用户数据。

## 11. 验收标准

- 四个页面可以通过桌面和手机导航切换，直接刷新子页面仍能打开。
- `Ctrl/Command + K` 在任何页面可用，并可通过键盘完成核心命令。
- 链接、记录和任务可以从万能收集箱创建、整理和删除，刷新后不丢失。
- 书签标签、收藏、搜索、筛选、访问记录和拖动排序可用。
- 任务可以新增、编辑、删除、安排、重复、完成和恢复。
- 专注可以关联任务，完成后进入最近七天统计。
- GitHub 和状态中心具备加载、成功、缓存、离线、限流和错误状态。
- 原有搜索、书签、主题、备份、登录、恢复和同步功能继续工作。
- 新设备登录后即使没有本地待上传数据，也能拉取云端内容。
- 桌面、平板和手机无横向滚动；移动触控目标和键盘焦点可用。
- 页面刷新后数据保留；登录用户新增实体可以跨设备同步。
- `pnpm lint`、`pnpm typecheck`、`pnpm test`、`pnpm build`、`pnpm e2e` 全部通过。
- 生产域名、API、静态资源、PWA、账户和同步通过公网验收。
- Git 仓库、线上部署和桌面备份指向同一意图版本，交付材料不含秘密。

## 12. 本次不做

- 插件市场或任意第三方小组件运行时。
- 在浏览器中保存私人 GitHub Token。
- 私人邮箱、私人日历、股票、新闻和热搜聚合。
- 在线 AI 自动分类或摘要。
- 完整网页、视频或 PDF 存档服务。
- 与四阶段无关的大范围视觉重做或技术栈替换。
